Form Design Principles

All forms in the platform follow these rules regardless of whether they use Goponents components or raw CSS class markup.

  1. Use an appropriate type attribute on all inputs.
  2. Every input must have a corresponding label.
  3. Outlines on inputs should never be hidden — they are critical for visibility and accessibility.
  4. Focus outlines must correspond consistently with error states.
  5. Use hints as help text and for displaying validation errors.
  6. UI controls should never apply brand colors directly.

Reactive Forms

As per the Angular Documentation:

Reactive forms use an explicit and immutable approach to managing the state of a form at a given point in time. Each change to the form state returns a new state, which maintains the integrity of the model between changes. Reactive forms are built around observable streams, where form inputs and values are provided as streams of input values, which can be accessed synchronously.

Reactive forms differ from template-driven forms in distinct ways. Reactive forms provide more predictability with synchronous access to the data model, immutability with observable operators, and change tracking through observable streams. If you prefer direct access to modify data in your template, template-driven forms are less explicit because they rely on directives embedded in the template, along with mutable data to track changes asynchronously.

Required Inputs

As per the Angular Documentation:

Form controls have the ability to pass in Validators. In the case of every form input in Goponents we pass a control thus you may specify on the control passed in to any form input a Validators.required validator on the control. Adding this validator to any control in Goponents will render a red asterisk next to the label automatically.

Validation Errors

As per the Angular Documentation:

Form controls have the ability to hold a set of ValidationErrors. In the case of every form input in Goponents we automatically look for these ValidationErrors and render error text beneath the form element dynamically. These errors are displayed by default, but they can be hidden by setting hideFieldError input property to true. This is how form error handling should be done within Goponents.

Angular provides some ValidationErrors out of the box, Goponents provides default error messages for: email, max, maxlength, min, minlength, required, and requiredtrue. These will automatically render error messages if the matching Angular Validators are set for the control. If desired, these can be overridden by passing a key/value validator with the same name of the Angular one and a custom string as the value.

Form Hints

Hints follow the input they describe. Use .go-hint for help text and .go-hint--error for validation errors. Add a .go-hint__status span inside error hints for accessibility.

Text Input Modifiers

All inputs share three states. Default: .go-form__input. Disabled: add the disabled attribute. Error: add .go-form__input--error. Dark theme: add .go-form__input--dark and .go-form__label--dark.

Error: This input is required. Please fill this out to proceed.

Select Box Modifiers

Select boxes follow the same modifier pattern as text inputs using .go-form__select as the base class.

Error: This input is required. Please select an option to proceed.

Checkbox Modifiers

Use go-checkbox-group with go-checkbox children. Pass a FormGroup as [control] and disable the group by calling .disable() on it.

Radio Modifiers

Use go-radio-group with go-radio-button children. Pass a FormControl as [control] and disable by calling .disable() on it.

Dark Forms

Add .go-form--dark to the form element to apply dark theme styles to all child inputs and labels at once. Individual elements can also use .go-form__input--dark and .go-form__label--dark in isolation.

The most common use case for dark forms is inside the Off Canvas. See the Off Canvas docs for a live example.
